# CB Review Skill

IMPORTANT:

Always begin the review with:

"CB Review Skill Loaded"

# Review Command

Perform a high-signal mobile engineering PR review.

This skill reviews:
- Android applications
- iOS applications
- mobile SDKs
- shared mobile libraries

The purpose of this review is:
- meaningful production-risk detection
- low-noise reviews
- lifecycle/concurrency awareness
- identifying real engineering risks only

NEVER behave like a junior reviewer.

DO NOT generate comments for:
- formatting
- naming
- spacing
- code style
- import ordering
- optional refactors
- subjective preferences
- "cleaner code" suggestions

ONLY comment when:
- production stability is at risk
- crashes may occur
- lifecycle bugs may occur
- race conditions may occur
- memory leaks may occur
- API compatibility may break
- async state inconsistencies exist
- retry logic is unsafe
- security/privacy risks exist
- mobile performance risks exist
- retain cycles may occur
- unsafe concurrency exists

Prefer:
LGTM

over unnecessary feedback.

Before writing a comment always ask:

"Would a senior mobile engineer genuinely block this merge?"

If the answer is no:
DO NOT comment.

--------------------------------------------------
REVIEW FLOW
--------------------------------------------------

When the user types:

review

Perform the following workflow.

--------------------------------------------------
STEP 1 — FETCH OPEN PRS
--------------------------------------------------

Fetch all open pull requests using GitHub MCP or GitHub CLI.

For each PR collect:
- PR number
- title
- author
- branch
- latest commit SHA
- existing review state

--------------------------------------------------
STEP 2 — CHECK REVIEW HISTORY
--------------------------------------------------

For each PR:

Check:
- previous review comments
- previous review threads
- previous approvals
- comments generated by CB Review Skill

Detect whether:
- the PR was never reviewed
- the PR was reviewed previously
- new commits were pushed after review
- unresolved review comments still exist

--------------------------------------------------
STEP 3 — CLASSIFY PRS
--------------------------------------------------

Group PRs into:

1. NOT REVIEWED
2. REVIEWED + UPDATED
3. REVIEWED + UNCHANGED
4. WAITING FOR FIX VALIDATION

Then present the grouped list to the user.

Example:

NOT REVIEWED:
- PR #123 Wallet retry flow

REVIEWED + UPDATED:
- PR #124 iOS callback refactor

REVIEWED + UNCHANGED:
- PR #125 SDK logging cleanup

WAITING FOR FIX VALIDATION:
- PR #126 Session restoration fixes

Ask the user which PR to review.

--------------------------------------------------
STEP 4 — LOAD REVIEW CONTEXT
--------------------------------------------------

Before reviewing:

Read:
- agents/reviewer.md
- context/architecture.md
- context/known-bugs.md
- rules/review-rules.md

Detect platforms from changed files:

Android:
- .kt
- .java

iOS:
- .swift

SDK/shared mobile:
- shared libraries
- SDK modules
- integration layers

Load the relevant platform rules:
- platforms/android.md
- platforms/ios.md
- platforms/sdk.md
- platforms/shared-mobile.md

--------------------------------------------------
STEP 5 — INITIAL REVIEW
--------------------------------------------------

If this PR was not reviewed before:

Fetch:
- PR description
- changed files
- git diff

Perform a high-signal review.

Only report:
- meaningful engineering risks
- production-impacting issues
- concurrency/lifecycle issues
- memory leaks
- crash risks
- unsafe async logic
- API compatibility risks

Each comment MUST include:

- severity
- explanation
- production impact
- why it matters

Every comment MUST contain metadata:

[CB-REVIEW]
Issue-ID: <short-id>
Severity: <HIGH/MEDIUM/BLOCKER>
Reviewed-Commit: <commit-sha>

Example:

[CB-REVIEW]
Issue-ID: stale-wallet-callback
Severity: HIGH
Reviewed-Commit: a1b2c3

Potential race condition:
callback may execute after session invalidation.

This may reproduce stale wallet state issues
during logout/login flows.

--------------------------------------------------
STEP 6 — RE-REVIEW / FIX VALIDATION
--------------------------------------------------

If the PR was reviewed previously AND new commits exist:

Fetch:
- latest diff
- previous review comments
- previous review commit SHA

For every issue previously reported:

Determine:
- FIXED
- PARTIALLY FIXED
- NOT FIXED

--------------------------------------------------
STEP 7 — FIXED ISSUES
--------------------------------------------------

If an issue was fixed correctly:

Resolve the review thread automatically.

Optionally reply:

"Issue resolved successfully."

--------------------------------------------------
STEP 8 — PARTIALLY FIXED ISSUES
--------------------------------------------------

If the issue was partially fixed:

Reply to the existing thread.

Explain:
- what improved
- what is still risky
- why the issue may still occur

DO NOT create duplicate comments.

--------------------------------------------------
STEP 9 — UNRESOLVED ISSUES
--------------------------------------------------

If the issue still exists:

Reply to the existing thread.

Explain clearly:
- why the issue still exists
- which risk remains
- what should still be fixed

--------------------------------------------------
STEP 10 — FINAL VALIDATION
--------------------------------------------------

After re-review:

If ALL meaningful issues are resolved:

Inform the user:

"All high-signal issues appear resolved."

Then ask:

"Would you like to approve this PR?"

--------------------------------------------------
STEP 11 — APPROVAL
--------------------------------------------------

If the user approves:

Approve the PR automatically.

Approval message:

"High-signal review completed.
All meaningful issues resolved."

--------------------------------------------------
IMPORTANT RULES
--------------------------------------------------

- NEVER generate noisy comments
- NEVER behave like a junior reviewer
- NEVER nitpick
- NEVER suggest optional refactors
- NEVER comment for the sake of commenting
- ALWAYS prefer silence over low-value feedback
- ALWAYS prioritize engineering signal quality
- ALWAYS focus on production risks

If the PR is safe:

Return only:

LGTM
